# 回复对象与输出路由

本文件用于所有 Agent 的通用路由。先判断**回复对象**，再判断渠道和格式。不要只因为原文是英文就直接写英文外发邮件。

## 1. 优先判断回复对象

| 用户表达 | 回复对象 | 默认输出 |
| --- | --- | --- |
| 「解决客户问题」「怎么看」「怎么处理」 | 用户内部 | 翻译/总结/依据/内部建议/待确认，不直接外发 |
| 「写回复客户的邮件」「编辑客户回复」「直接回客户」 | 客户 | 外发英文或中文客户话术；英文外发后附完整中文翻译 |
| 「淘宝」「天猫」「买家」 | 买家 | 短、口语、可复制；少解释，不写长分析 |
| 「发销售」「私有邮件」「内部转交」 | 销售/内部团队 | 只写对方需要做什么、需要给什么信息 |
| 「帮我问同事/合作伙伴/那边」 | 同事/合作伙伴 | 用第一人称内部询问语气，不写「客户想问」除非用户要求 |
| 「这能不能这么说」「这句话对吗」 | 用户内部 | 判断是否合适 + 给更好的改写 |
| 「结案」「沉淀」「记住」 | 仓库 | 按知识库/记忆/案例规则更新并开 draft PR |

如果回复对象不清楚，先按**用户内部建议**输出；不要直接生成客户可见回复。

## 2. 英文邮件默认禁止直接外发

除非用户明确要求「写外发」「回复客户」「编辑邮件」「直接发客户」，英文客户邮件默认输出：

```text
完整中文翻译 → 问题总结 → 依据资料 → 内部处理建议 → 需转交内容 → 待确认事项 → 结论置信度
```

外发英文邮件只在用户确认方向或明确要求后生成。外发正文要短，结束后附完整中文翻译。

## 3. 输出长度

| 场景 | 长度与风格 |
| --- | --- |
| 淘宝/天猫 | 1-3 段，短句，口语，可复制；一般不写来源和置信度标签 |
| 用户内部快速问答 | 先给结论，再给必要解释；能一句说清不展开 |
| 英文邮件内部建议 | 可完整，但结构固定，置信度给用户看 |
| 英文外发客户 | 3-6 个短段，回答问题即可，避免长铺垫 |
| 销售/内部转交 | 5-8 行清单，直接说明客户需要什么、对方要提供什么 |
| 同事/合作伙伴询问 | 1 个自然段或 2-4 个 bullet，用第一人称 |

## 4. 三种版本按需输出

复杂问题可按对象拆开，不要混在一起：

```text
内部判断：给用户看的事实、置信度、风险。
客户可发：可直接复制给客户的话术。
问同事/销售：内部协作口径，只写需要对方确认或提供的信息。
```

用户只要其中一种时，只输出对应版本。

## 5. 事实、推断、建议、禁止

涉及兼容性、性能、硬件设计、版本支持时，内部建议必须区分：

```text
事实：资料或内部确认已经明确的内容。
推断：基于事实的合理判断，必须说明依据不足时不能当结论。
建议：推荐给客户的处理动作或话术。
禁止：不能承诺、不能混用、不能外发的说法。
```

客户可见回复中不必逐字写这些标签，但 Agent 内部判断必须有边界。

## 6. 问问题的数量

给客户补充信息时，优先问最影响判断的 1-3 项。确实需要更多信息时，先内部说明原因，再把客户可见问题压缩成最关键几项。

## 7. 快速模式

当用户说「怎么回」「给一句」「问那边」「发销售」「淘宝客户问」时，默认给最短可复制版本。高风险技术问题可以先给一句结论，再补内部注意事项。不要用长分析淹没可复制答案。

## 8. 用户纠正优先

如果用户纠正「不是邮件」「这是淘宝」「这是给销售」「换成我问问题的语境」，立即按新对象重写。不要继续解释旧答案，除非用户问原因。
